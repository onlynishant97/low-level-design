Tracks real-time availability of each seat on a flight — the source of truth for seat state and the primary concurrency battleground.

🔒 CONCURRENCY CRITICAL
How Concurrency is Handled
⚡ Optimistic Locking: The version field is incremented on every state change (AVAILABLE→LOCKED→BOOKED). Any stale write is detected and rejected before persisting.
🗂️ ConcurrentHashMap: The repository backing this model uses ConcurrentHashMap ensuring thread-safe reads and writes at map level.
🔑 Composite Key: Stored under key flightId:seatNumber enabling atomic per-seat operations without locking unrelated seats.
SeatLock
Represents a temporary exclusive hold on a seat during the booking initiation window — prevents double booking while payment is in progress.
Fields
- flightId : String
- seatNumber : String
- bookingId : String
- expiresAt : LocalDateTime  ⚡ TTL
Methods
+ isExpired() : boolean
🔒 CONCURRENCY CRITICAL
How Concurrency is Handled
⏱️ TTL-based expiry: expiresAt is set to now + 5 minutes. isExpired() provides a fast O(1) check before any lock decision.
⚛️ Atomic acquisition: Lock is acquired via ConcurrentHashMap.compute() — the check-and-set is a single atomic operation, eliminating the TOCTOU race condition.
🧹 Auto-cleanup: SeatLockExpiryService runs every 60s to remove expired locks and reset seats to AVAILABLE.

🔒 CONCURRENCY CRITICAL
How Concurrency is Handled
🔑 Per-seat locking: A ConcurrentHashMap<String,Object> stores one lock object per flightId:seatNumber key. Only the thread booking a specific seat is blocked — other seats on the same flight proceed in parallel.
🔄 Rollback on partial failure: If seat N locks successfully but seat N+1 fails, a catch block iterates all previously locked seats and calls unlockSeat() atomically — no ghost locks are left behind.
✅ State validation: Guards check BookingStatus before every transition — e.g., cannot confirm a CANCELLED booking, cannot cancel an already-CANCELLED booking.

SeatInventoryService
Manages per-seat state transitions with an in-memory lock map and ownership validation for safe concurrent access.
Dependencies
- repository : SeatInventoryRepository depends
- locks : ConcurrentHashMap<String, SeatLock> ⚡
- LOCK_DURATION_MINUTES = 5
Methods
+ lockSeat(flightId, seatNumber, bookingId)
+ confirmSeat(flightId, seatNumber, bookingId)
+ releaseSeat(flightId, seatNumber)
+ acquireLock(flightId, seatNumber, bookingId)
+ releaseLock(flightId, seatNumber)
+ validateLockOwner(flightId, seatNumber, bookingId)
🔒 CONCURRENCY CRITICAL
How Concurrency is Handled
⚛️ Atomic lock acquisition: acquireLock() uses ConcurrentHashMap.compute() — the entire check-existing + insert-new sequence is one atomic operation, eliminating the classic TOCTOU race condition.
👤 Ownership validation: validateLockOwner() verifies the bookingId on the lock matches the caller before allowing confirmation — prevents one booking from stealing another's seat.
⏱️ Expiry awareness: Every lock operation checks SeatLock.isExpired() first — expired locks are treated as non-existent, allowing new acquisitions.

SeatLockExpiryService
Background scheduler that runs every 60 seconds to detect and release expired seat locks — prevents ghost locks from blocking seats indefinitely.
Fields
- locks : Map<String, SeatLock> depends
- repository : SeatInventoryRepository depends
- scheduler : ScheduledExecutorService ⚡
Methods
+ start()
+ stop()
- releaseExpiredLocks()
🔒 CONCURRENCY CRITICAL
How Concurrency is Handled
⏰ Scheduled executor: Uses Executors.newSingleThreadScheduledExecutor() with scheduleAtFixedRate() — runs cleanup every 60 seconds in a dedicated background thread.
🧹 Atomic removal: Uses ConcurrentHashMap.entrySet().removeIf() — the predicate check + removal is atomic per entry, safe for concurrent lock map access.
🔓 Safe seat reset: Only resets seats in LOCKED state (not BOOKED) — prevents accidental freeing of confirmed reservations.
