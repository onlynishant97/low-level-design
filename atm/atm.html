<<model>> Account
Represents a bank account. Owns thread-safe debit and credit operations using a per-account lock.
Fields
- accountNumber: String
- balance: BigDecimal
- lock: ReentrantLock [concurrency]
Methods
+ debit(amount): void
+ credit(amount): void
+ getBalance(): BigDecimal
🔒 Concurrency Note
  • Each Account has its own ReentrantLock.
  • debit() and credit() acquire the lock before modifying balance, ensuring no two threads can modify the same account simultaneously.
  • Lock is released in a finally block to prevent deadlocks.

🔒 Concurrency Handling — Full Summary

Class Mechanism What It Protects
Account ReentrantLock Prevents concurrent debit/credit on the same account. Each account has its own lock for fine-grained concurrency.
CashManagementServiceImpl synchronized Ensures only one thread at a time can check or modify the ATM cash inventory. Prevents double-dispensing.
InMemoryAccountRepository ConcurrentHashMap Thread-safe account lookups and saves without explicit locking.
InMemoryCardRepository ConcurrentHashMap Thread-safe card lookups and saves without explicit locking.
InMemoryTransactionRepository ConcurrentHashMap + CopyOnWriteArrayList Thread-safe append-only transaction log. CopyOnWriteArrayList is ideal for read-heavy audit logs.
Atm volatile Ensures state transitions (currentState field) are immediately visible to all threads without caching stale state.
CashInventory ⚠️ None (intentional) Deliberately unsynchronized. CashManagementServiceImpl is the sole owner of the lock for cash operations.
